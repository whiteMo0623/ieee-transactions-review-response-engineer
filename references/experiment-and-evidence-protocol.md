## 1. 疑虑到实验的映射

为当前意见建立矩阵，每行包含：审稿人疑虑、被质疑的论文主张、现有证据、证据缺口、所需实验/分析、可观测量、结论规则。实验不得只展示“更多结果”；它必须区分至少两个与疑虑相关的解释。

优先选择最直接的证据形式：正确性问题用单元/合成验证与端到端结果；公平性问题用相同数据、预算和调参协议；泛化问题用预先定义的域外数据；鲁棒性问题用有语义的扰动强度曲线；效率问题同时报告质量、延迟、吞吐、峰值显存、参数量或 FLOPs，并给出测量环境。

在看到结果前写明判定规则。允许三类结果：支持原主张、部分支持并收缩边界、不支持并修改/撤回主张。

## 2. 完整性与公平性

从原论文或官方实现恢复正式训练预算，不以时间便利调整。对所有方法使用同一数据版本、划分、预处理、评价实现、硬件类别和 checkpoint 选择原则。若方法特性要求不同设置，解释其必要性并同时报告资源差异。

使用实验设计预先列出的全部随机种子。固定并记录 Python、NumPy、框架和数据加载器的随机源；记录确定性设置及已知非确定算子。不得只挑选成功或有利的种子。

基线优先使用审稿人点名方法、论文原有强基线和能隔离当前机制的控制组。使用作者官方实现或明确记录的可靠实现；不得给基线不公平的较弱预算或调参范围。

## 3. 复用判定

只有同时满足下列条件才可把既有结果作为当前意见的直接证据：研究问题一致；数据版本、划分与预处理一致；模型和代码提交一致；训练预算与 checkpoint 规则一致；对比方法和指标足够；种子及统计单位一致；原始结果、日志和配置可访问。

任一条件不满足时，将既有实验作为背景而非充分证据。可复用不变的数据管道、模型组件和绘图代码，但对当前意见新增的控制变量、数据、基线、指标和分析必须完整执行。

## 4. 执行与可复现记录

在 `experiments/manifest.json` 使用如下顶层结构：

```json
{
  "schema_version": 1,
  "planned_seeds": [0, 1, 2],
  "planned_runs": ["exp1-methodA-seed0", "exp1-methodA-seed1", "exp1-methodA-seed2"],
  "runs": [
    {
      "id": "exp1-methodA-seed0",
      "question_id": "Q1",
      "status": "completed",
      "seed": 0,
      "command": "python ...",
      "config": "experiments/configs/...",
      "log": "experiments/logs/...",
      "raw_result": "experiments/raw/...",
      "checkpoint": "...",
      "exit_code": 0,
      "started_at": "ISO-8601",
      "finished_at": "ISO-8601"
    }
  ]
}
```

`planned_runs` 必须在正式执行前写定，并覆盖方法、数据集、设置和种子的笛卡尔组合；完成状态按运行 ID 逐一核验，而不是仅检查“每个种子至少出现一次”。若当前意见确实不需要任何新实验，仅在疑虑不涉及新增经验性证据且现有材料已能直接回答时，将 `case.json` 的 `experiment_required` 设为 `false`，并在方案中给出可审计理由。不得把该字段作为规避耗时实验的开关。

为环境保存锁文件或完整包清单、框架/CUDA/驱动版本和硬件信息。记录数据集版本或校验值、代码 Git commit 与 dirty 状态。正式运行前通过接口、维度、指标和小型合成测试；测试结果不能替代正式结果。

原始输出保持不可变。所有清洗、聚合、统计和制图由 `experiments/src/` 中的脚本从原始结果生成，并记录命令。失败和中断的运行保留状态及日志，修复后新增运行记录，不覆盖历史。

## 5. 统计与结论门槛

明确实验单位和统计单位。以数据样本为单位的配对比较使用相同测试样本；以训练重复为单位的不确定性保留逐种子值。根据假设和分布选择预先规定的检验，报告效应量与置信区间，必要时进行多重比较校正。

不要把 `p > 0.05` 解释为等价。若要主张无实质差异，使用预先给定的等价界值和等价检验或置信区间。若结果混合，按数据集、场景或扰动强度限定结论，不用总体均值掩盖失败子组。

## 6. 证据账本

`response/evidence-ledger.json` 必须为对象数组。每个对象至少使用 `id`、`claim`、`source` 三个字段；`source` 是案件目录内的原始或处理后证据路径。涉及数值时再记录 `value`、JSON/CSV 字段或行列、`generator`、`config`、`run_ids` 及对应表/图编号。人工计算的数值必须有可执行生成脚本。

成文后反向审计每个比较级、因果词、显著性词和范围词。无法绑定证据的语句删除、改写为解释，或明确标注为假设。
